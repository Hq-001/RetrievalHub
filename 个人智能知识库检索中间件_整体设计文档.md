## 一、项目概述

### 1.1 项目定位

本项目构建一套**个人级智能知识库检索中间件（Retrieval Middleware）**，定位为 **RAG-Ready** 的检索基础设施：**只负责检索（R），不负责生成（G）**。

系统将"文档沉淀—知识切分—向量化入库—混合检索—结构化返回"打通为闭环，对上层以两种方式暴露能力：

1. **RESTful API**：仅用于文档管理（上传、删除、列表）。
2. **MCP Server（Model Context Protocol）**：将检索能力封装为 `search_knowledge` 工具，供外部 AI Agent（如 Codebuddy、Claude、各类 LLM 客户端）直接调用，返回**结构化 Chunk 列表**（含原文与来源元数据），**不生成摘要、不生成回答**。

系统的核心价值在于：

- **专注检索、职责单一**：不内置 LLM，不做问答生成，把生成交给最适合的外部 Agent，自身做到"检索又快又准"。
- **RAG-Ready**：以标准 MCP 协议对接任意 AI Agent，即插即用，成为 Agent 的"知识外脑"。
- **幻觉责任明确转移**：中间件只返回有出处的原文片段，是否据此生成、是否产生幻觉，责任在调用方 Agent。这一点在 MCP 工具描述中显式声明。
- **解析零风险**：仅支持结构化/半结构化文本格式（Markdown、JSON），从根源消除 PDF/OCR 的解析不确定性。

### 1.2 定位变更对比（v1.0 → v2.0）

| 维度 | v1.0（独立 RAG 应用） | v2.0（检索中间件） |
| --- | --- | --- |
| 系统职责 | 检索 + 生成（R + G） | 仅检索（R） |
| 生成能力 | 内置 LLM 问答 | **删除**，交外部 Agent |
| 查询改写 | 内置 | **删除**（由 Agent 侧完成） |
| 会话管理 | 内置多轮对话 | **删除**（无状态检索） |
| 对外接口 | REST（文档+问答+会话） | REST（仅文档管理）+ **MCP Server** |
| 输入格式 | PDF/MD/TXT（含 OCR） | **仅 MD + JSON** |
| 幻觉责任 | 系统自身抑制 | **转移给调用方 Agent** |
| 交付形态 | 完整应用 | 可嵌入的检索中间件 |

### 1.3 设计原则

| 原则 | 说明 |
| --- | --- |
| 职责单一 | 只做检索，不越界做生成；生成交给外部 Agent |
| 模块解耦 | 各能力职责单一，通过明确接口协作 |
| 配置外置 | 所有密钥、模型参数、权重系数、路径通过 .env 统一管理，零硬编码 |
| 可插拔 | 嵌入模型、重排序模型、检索策略均可替换，面向接口编程 |
| 解析可靠 | 仅支持结构化文本格式，解析成功率 ≥ 99.9% |
| 零停机迁移 | 模型升级采用双集合写入，旧集合只读，验证后异步清理 |
| 可观测 | 全链路结构化日志与关键指标埋点 |
| 安全优先 | 私有文档本地存储，密钥不落盘明文 |

### 1.4 名词约定

- **文档（Document）**：用户上传的一个完整文件（.md / .json / .jsonl），对应唯一文档 ID。
- **分块（Chunk）**：文档切分后的最小检索单元，携带 content_type、section_path 等结构化元信息。
- **知识库（Knowledge Base）**：一组文档及其向量/全文索引的集合，逻辑隔离。
- **混合检索（Hybrid Search）**：向量召回 + 全文（BM25）召回并融合。
- **重排序（Rerank）**：对融合结果做 cross-encoder 精排，**仅作用于检索结果，不参与生成**。
- **MCP（Model Context Protocol）**：AI Agent 与外部工具/数据源交互的标准协议，本系统以 MCP Server 形式暴露检索工具。
- **双集合写入（Dual-Collection Write）**：模型升级时新旧集合并存的零停机迁移策略。

## 二、系统总体架构

### 2.1 分层视图

系统采用四层架构：**接入层、应用层、能力层、存储层**。注意接入层由 **REST（文档管理）** 与 **MCP Server（检索）** 双通道构成。

```
┌──────────────────────────────────────────────────────────────┐
│  接入层                                                        │
│   ├─ RESTful API：文档管理（上传 / 删除 / 列表）              │
│   └─ MCP Server（stdio / SSE）：search_knowledge 检索工具     │
├──────────────────────────────────────────────────────────────┤
│  应用层：文档管理服务 / 检索服务 / 知识库管理服务             │
├──────────────────────────────────────────────────────────────┤
│  能力层：MD 解析 / JSON 解析 / 分块 / 嵌入 / 混合检索 / 重排序 / 热点缓存(LRU)│
├──────────────────────────────────────────────────────────────┤
│  存储层：向量+全文索引(LanceDB) / 元数据库 / 原文存储         │
└──────────────────────────────────────────────────────────────┘
```

> 相比 v1.0，能力层删除了 **LLM、查询改写、问答生成**；接入层删除了 **/chat、/sessions**，新增 **MCP Server**；存储层删除了 **对话历史**。

### 2.2 架构说明

**接入层（双通道）**
- **RESTful API**：面向"数据面运维"，负责文档的上传、删除、列表，承担鉴权、限流、请求日志。版本化前缀 `/v1/`。
- **MCP Server**：面向"AI Agent 调用"，通过 stdio 或 SSE 两种传输方式暴露 `search_knowledge` 工具。Agent 传入查询与检索参数，返回结构化 Chunk 列表。**不生成任何自然语言回答**。

**应用层**
- 文档管理服务：接收上传后**异步**编排"解析（MD/JSON）→ 分块 → 批量嵌入 → 写入索引"，立即返回 `202 Accepted` 与状态查询地址；客户端轮询获取最终状态（`processing` / `ready` / `failed`）。
- 检索服务：编排"热点缓存命中 → 混合召回 → 动态加权融合 → 重排序 → 结构化组装"，是系统核心。**无查询改写、无 LLM 生成**。
- 知识库管理服务：知识库的创建、列出、删除等元数据操作。

**能力层**
- 每个能力以"接口 + 默认实现"形式存在。删除了 LLM 抽象、查询改写、问答生成三个模块。

**存储层**
- 推荐 **LanceDB**：原生支持向量 + 全文（BM25/FTS）混合检索，避免向量索引与 BM25 索引分离带来的一致性问题。
- 元数据库（SQLite）：文档/分块的元信息。
- 原文存储：保留原始文件用于溯源展示。
- **删除对话历史存储**（无会话概念）。

### 2.3 数据流总览

**写入流（文档入库，202 Accepted + 状态轮询）**：
上传文件（.md/.json/.jsonl）→ 格式校验（扩展名 + 内容嗅探）→ **立即返回 202 Accepted**（含文档 ID 与状态查询地址）→ 后台异步执行：解析（MD:AST / JSON:扁平化）→ 分块（MD:递归标题继承 / JSON:键值原子化）→ 批量嵌入 → 写入 LanceDB（向量 + 全文索引）+ 元数据库 → 文档状态由"处理中"置为"可检索"或"失败"。

> 采用**异步入库（202 + 状态轮询）**：上传接口不再阻塞等待解析与嵌入，立即返回 `202 Accepted` 与文档状态查询地址，客户端通过 `GET /v1/documents/{doc_id}` 轮询状态（`processing` / `ready` / `failed`）。这在不显著增加首版复杂度的前提下，彻底消除大文件同步阻塞与 504 超时痛点。底层以 `EnqueueBackend` 抽象实现，默认走**进程内后台任务（asyncio.Task / 线程池）**，零外部依赖；后续可平滑替换为 RQ（Redis Queue）以支撑大规模批量场景。失败文档仍支持 `POST /v1/documents/{doc_id}/retry` **手动重试**（幂等，按内容哈希去重）。

**读取流（知识检索，无生成）**：
Agent 调用 `search_knowledge`（含查询 + 检索参数）→ **热点查询 LRU 缓存命中则直接返回** → 否则混合召回（向量 + BM25 并行）→ 按 content_type 动态加权融合 → 重排序精排 → 组装结构化 Chunk 列表（原文 + 来源元数据）→ 写入缓存并返回给 Agent。**到此结束，不拼 Prompt、不调 LLM、不生成回答。**

## 三、核心模块设计

### 3.1 文档管理模块

**职责**：文档全生命周期管理——上传、解析、分块、入库、列表、删除。

**上传与校验**
- **仅支持** Markdown（.md）与 JSON（.json / .jsonl）。
- 双重校验：文件扩展名 + 内容嗅探（尝试 UTF-8 解码；JSON 尝试 `orjson` 解析验证结构合法性）。
- 限制单文件大小（默认 ≤ 10MB）与并发上传数。
- **彻底移除** PDF、TXT、OCR 相关依赖与代码路径，从根源消除解析风险。

**异步入库流水线（202 Accepted + 状态轮询）**
- 上传即返回 `202 Accepted`（含文档 ID 与状态查询地址），解析、分块、批量嵌入、写索引在**后台异步**完成，客户端通过状态接口轮询进度，彻底消除大文件同步阻塞。
- 批量嵌入以降低嵌入调用次数与延迟。
- 文档状态机：`processing`（处理中）→ `ready`（可检索）/ `failed`（失败），状态可由 `GET /v1/documents/{doc_id}` 查询。
- 失败可经 `POST /v1/documents/{doc_id}/retry` **手动重试**（幂等，按内容哈希去重）；重试复用同一后台流水线。
- 底层通过 `EnqueueBackend` 抽象解耦：首版默认**进程内后台任务**（asyncio 任务 / 线程池），零外部依赖；后续可平滑切换为 RQ（Redis Queue）以支撑高并发批量写入。
- 后台任务设总超时上限（默认 30s，`.env` 项 `INGEST_TIMEOUT_SEC`），超时后标记文档状态为 `failed` 并记录原因（因上传已先返回 202，不再返回 504）。
- **崩溃恢复（进程内任务易失性对策）**：SQLite 为 `processing` 文档记录 `started_at` 与 `attempt_count`。系统启动时扫描"仍 `processing` 且已超过 `INGEST_TIMEOUT_SEC * 2`"的文档，自动置为 `failed` 并记 `reason: "process_restarted"`，引导用户手动重试。后期切换 RQ（任务持久化于 Redis）后该风险自然消失。

**文档元信息**
- 字段见 §6。新增 `format_version`（解析器格式版本），用于解析器升级后的兼容管理与按版本重建。

**去重与增量**
- 内容哈希识别重复上传；替换文档时先删旧分块与向量再入新分块。
- **并发上传竞态防护**：数据库层对 `(knowledge_base_id, content_hash)` 建立 **UNIQUE 约束**。两请求同时上传相同内容时，先到者写入元数据，后到者插入触发唯一冲突，返回 **`409 Conflict`** 并附带已存在的文档 ID，原子性完成去重，无需依赖分布式锁。

### 3.2 Markdown 解析与分块模块

**职责**：将 Markdown 解析为结构化分块，保留标题层级语义。

**AST 解析**
- 采用 `markdown-it-py` 将 MD 解析为 **AST（抽象语法树）**，而非正则处理。
- 从 AST 提取：标题层级（H1~H6）、代码块（含语言标识 `code_language`）、段落、列表。

**递归标题继承法分块**
- 以 `#`（H1）和 `##`（H2）作为**强制切分边界**：遇到 H1/H2 即开启新块。
- 每个块自动继承其所属的标题路径，生成 **`section_path`** 元数据，如 `"第三章 > 注意力机制"`。
- H3 及更深标题不强制切分，但 `section_path` 最多继承至 H3 级别（即深度 ≤ 3），超出部分压平到 H3 层级承载，防止元数据过长（`section_path` 示例：`"第三章 > 注意力机制 > 多头机制"`，H4+ 内容仍落入该 H3 块内，不再下钻）。
- **代码块强制完整**：任何情况下不在代码块内部切分，代码块整体归入所在分块；若单个代码块超长，独立成块并保留 `code_language`。
- **嵌套列表继承深度限制 ≤ 3 级**：防止深层嵌套列表把内容切得过碎（分块碎片化）；超过 3 级的层级被压平到第 3 级承载。

**分块参数（MD 独立配置）**
- Markdown 使用独立前缀 `MD_*`：块大小上限 `MD_CHUNK_SIZE`（默认 800，单位字符）、重叠量 `MD_CHUNK_OVERLAP`（默认 100，单位字符），均通过 .env 配置，默认面向通用叙述性文本场景调优。
- 与 JSON 的分块参数解耦（见 §3.3），便于针对"叙述性文本"与"键值原子数据"分别调优，互不影响。

### 3.3 JSON 解析与分块模块

**职责**：将 JSON / JSONL 解析为结构化分块。

**扁平化压平（Flatten）**
- 采用 `orjson` 高性能解析。
- 嵌套对象**压平为原子键值对**，键用点号路径表示，如 `{"a":{"b":1}}` → `"a.b: 1"`。
- 每个原子键值对（或语义相关的一组键值对）成为可检索文本。

**数组与父子关系**
- 数组项（典型如 FAQ 列表、条目列表）**独立拆分为多个 Chunk**，同一数组下的各项**共享 `json_parent_id`**（指向数组所在的父路径），保留归属关系，便于溯源与聚合。
- **`json_parent_id` 生成规则**：取数组所在 **JSON Path**（如 `$.faq_list`、`$.data.items`）的 **MD5 哈希值**作为 `json_parent_id`。以路径而非自增序号为基准，确保同一数组下所有子块拥有**稳定且一致**的 parent_id（与文档内容无关、可跨次入库复现），便于按父聚合检索与溯源。
- JSONL 每行作为一个独立顶层对象处理。

**超长 Value 降级**
- 当某个 Value 超长（> 2000 tokens，阈值可配）时，**降级为 Markdown 语义切分**：对该长文本按 §3.2 的语义规则二次切分为多个子块，子块共享同一 `json_parent_id` 与键路径。

**分块参数（JSON 独立配置）**
- JSON 使用独立前缀 `JSON_*`：块大小上限 `JSON_CHUNK_SIZE`（默认 600，单位字符）、重叠量 `JSON_CHUNK_OVERLAP`（默认 60，单位字符），均通过 .env 配置。
- 由于 JSON 已扁平化为原子键值对，单块通常较短；该参数主要用于控制"语义相关键值对分组"的上限，与 MD 的解耦便于针对性调优。

### 3.4 嵌入与向量化模块

**职责**：文本分块 → 向量；查询 → 向量。

**嵌入模型抽象**
- 统一嵌入接口（文本 → 向量），实现可插拔。
- 默认实现通过 .env 配置 Endpoint / Key / 模型名 / 维度 / 批量大小（不绑定特定厂商，OpenAI 兼容协议即可；已从依赖中删除 `openai` 硬依赖，改为可选适配器）。
- 预留本地嵌入模型实现位（如 sentence-transformers 系列）。

**批量化与维度管理**
- 入库时批量嵌入，降低调用次数与延迟。
- 知识库内向量维度必须一致；切换模型触发**双集合迁移**（见 §8.3）。

### 3.5 存储模块（LanceDB）

**职责**：持久化向量与全文索引，支持高效混合检索。

**选型：LanceDB**
- **原生支持向量检索 + 全文检索（FTS/BM25）于同一存储**，避免"向量库 + 独立 BM25 索引"双索引的**一致性问题**（新增/删除文档时两套索引易失步）。
- 列式存储、本地持久化、零外部服务依赖，单机即可运行。
- 支持按元信息过滤（content_type、文档 ID、知识库 ID 等）与版本化。

**索引组织**
- 每个知识库对应独立集合（collection），逻辑隔离、高效删除。
- 向量与元信息（含原文片段）关联存储，检索结果直接带回溯源所需字段。
- 抽象"存储接口"，后续可平滑替换为 Milvus / Qdrant + 独立 FTS（如需）。

**存储加密（基础支持）**
- 引入 `ENABLE_STORAGE_ENCRYPTION` 开关（`.env`，**默认关闭**），为后续"透明加密（Encryption at Rest）"预留基础能力，首版不增加过多复杂度。
- 开关开启时，原文存储与（可选）向量/元数据库落盘前经由统一加密层处理；首版仅落地开关与加解密接口位，默认不启用。
- 密钥通过 `.env` 注入（如 `STORAGE_ENCRYPTION_KEY`），禁止硬编码与明文落盘；具体算法（AES-256-GCM 等）在后续阶段实现。

### 3.6 检索模块（系统核心）

**职责**：从知识库高效、精准召回相关内容，返回结构化 Chunk。采用**混合召回 → 动态加权融合 → 重排序**三级管道。**无查询改写、无生成。**

**第一级：混合召回**
- **向量召回**：语义相似度检索，擅长同义/近义。
- **全文召回（BM25）**：LanceDB FTS，擅长专有名词、编号、代码标识符、JSON 键名。
- 两路各取 top-N 候选进入融合。

**第二级：按 content_type 动态加权融合**
- 基础融合采用 RRF（Reciprocal Rank Fusion）。
- **按 content_type 差异化加权**（本版新增核心策略）：
  - **MD 块**：向量得分 × `MD_VECTOR_WEIGHT`（默认 1.1）——Markdown 是叙述性文本，语义检索更有效。
  - **JSON 块**：BM25 得分 × `JSON_BM25_WEIGHT`（默认 1.2）——JSON 多为键值/枚举/编号，精确匹配更有效。
- 权重系数通过 .env 开放配置（`MD_VECTOR_WEIGHT` / `JSON_BM25_WEIGHT`），零硬编码。

**第三级：重排序（Rerank）**
- 对融合候选调用 cross-encoder 重排序模型做 query-chunk 精排。
- **仅作用于检索结果排序，绝不参与生成**。
- 重排序模型抽象为接口，可替换云端/本地实现，可通过 .env 关闭以降低延迟。

**热点查询缓存（LRU）**
- 引入轻量级本地缓存（LRU，如 `functools.lru_cache` / `cachetools`）缓存检索结果（含重排序输出）。
- 同一知识库被频繁检索相同 query 时直接命中缓存，跳过召回/融合/重排序计算，**显著减少重排序（cross-encoder）调用次数并降低端到端延迟**。
- **精细失效（避免全库清空）**：缓存键由 `(kb_id, query, params)` 扩展为 `(kb_id, doc_versions_hash, query, params)`。维护知识库下所有 `(doc_id, last_modified)` 的全局哈希 `doc_versions_hash`；文档变更时仅更新该哈希，使**仅依赖被改文档的查询结果因哈希失配而自动驱逐**（按哈希分区），而非清空整个知识库命名空间，缓存命中率不受单文档更新影响。若实现复杂，可退化为 **TTL 为主（`QUERY_CACHE_TTL_SEC`，默认 300s）+ 主动失效为辅**的兜底方案。容量/过期通过 .env 配置（`QUERY_CACHE_MAXSIZE` / `QUERY_CACHE_TTL_SEC`）。

**评测集与权重自动调优**
- 构建**查询-相关性标注集**（见 §9.1 / §11 阶段二）：按 content_type **分层抽样**覆盖多样样本——MD 含 3 种子类型（纯文本 / 代码块 / 嵌套列表），JSON 含 3 种子类型（键值对 / 数组对象 / 混合），并以人工/半自动标注的 `query → 相关 Chunk` 对作为回归基线。
- 引入**自动调参工具**：以标注集为优化目标，对动态加权系数（`MD_VECTOR_WEIGHT` / `JSON_BM25_WEIGHT`）、召回 top-N、重排序 top-K 等超参做网格/贝叶斯搜索。
- **防过拟合**：自动调参采用 **K-Fold 交叉验证**，并预留 **20% 盲测集（hold-out）** 用于最终验证，确保权重在未见样本上仍具泛化能力，而非仅在评测集上过拟合。最优参数组合建议写入 `.env` 或配置基线。

**可插拔管道**
- 建模为"召回器 → 融合器 → 重排器（+ 缓存）"接口链，新增算法只需实现接口并注入。

### 3.7 MCP Server 模块（对外检索入口）

**职责**：以 MCP 协议将检索能力暴露为工具，供外部 AI Agent 调用。

**传输方式**
- 支持 **stdio**（本地进程内嵌，如桌面 Agent 客户端）与 **SSE**（远程/网络调用）两种传输。
- **SSE 鉴权与连接管理**：SSE 暴露于网络时，在**握手阶段（SSE 连接建立）及每次工具调用**支持传递 `Authorization` 头或 API Key 参数，**复用 REST 层鉴权逻辑**，避免未授权调用；stdio 模式因进程内嵌默认信任，无需额外鉴权。连接断开后由客户端按 MCP 协议重连，网络抖动可结合 §3.7 超时/重试约定处理。

**工具定义：`search_knowledge`**
- 入参：`query`（查询文本）、`knowledge_base_id`、`top_k`（可选）、`rerank`（是否重排序，可选）、`filters`（可选，如按 content_type 过滤）、`timeout_ms`（可选，单次检索超时，默认 5000，见下）。
- 出参：**结构化 Chunk 列表**，每个元素含：
  - `text`：分块原文（不做摘要、不改写）
  - `score`：相关性得分
  - `source`：来源元数据（文档名、content_type、section_path 或 json_parent_id、code_language、字符偏移等）
- **明确不做**：不生成摘要、不生成回答、不调用任何 LLM。
- **超时与重试（写入工具描述）**：工具描述中**明确推荐超时时间与重试策略**，例如：
  > "建议客户端设置单次调用超时 **5s（`timeout_ms` 默认 5000）**；若超时或返回临时错误（5xx/网络抖动），建议**指数退避重试（最多 3 次，初始间隔 200ms）**。本工具为只读检索，重试安全且幂等。"
- 允许 Agent 通过 `timeout_ms` 入参**显式控制单次检索超时**（覆盖默认），便于在不同延迟要求下权衡召回质量与响应速度。

**幻觉责任声明（写入工具描述）**
- 工具描述中显式声明：
  > "本工具仅返回知识库中检索到的原始文本片段及其来源，不生成任何结论或回答。基于这些片段生成内容的准确性与幻觉风险，由调用方 AI Agent 承担。"
- 通过协议层显式**将幻觉责任转移给外部 AI Agent**。

## 四、接口设计

### 4.1 RESTful API（仅文档管理）

> 遵循 RESTful 风格与版本化前缀 `/v1/`。**已移除 /chat、/sessions。**

- **上传文档**：`POST /v1/documents`
  - 入参：文件（multipart，仅 .md/.json/.jsonl）、目标知识库 ID、可选元信息。
  - 出参：`202 Accepted` + 文档 ID、上传时间、初始状态（`processing`）、状态查询地址、format_version。
  - 行为：**异步**解析、分块、批量嵌入、写索引；接口**立即返回 202**，不阻塞等待；客户端轮询文档详情获取最终状态。

- **查询文档列表**：`GET /v1/knowledge-bases/{kb_id}/documents`
  - 支持分页、按状态过滤。

- **查询文档详情**：`GET /v1/documents/{doc_id}`
  - 返回元信息、分块数量、解析状态（`processing` / `ready` / `failed`）、format_version，兼作状态轮询入口。

- **删除文档**：`DELETE /v1/documents/{doc_id}`
  - 级联删除其全部分块与向量。

- **重试入库**：`POST /v1/documents/{doc_id}/retry`
  - 手动重试失败文档，幂等（按内容哈希）。

- **健康检查（存活）**：`GET /healthz`
  - 返回 `200 OK` 表示进程存活（只读、不依赖外部资源），供 K8s `livenessProbe` / Docker `HEALTHCHECK` 使用。

- **就绪探针（就绪）**：`GET /readyz`
  - 返回 `200 OK` 表示服务就绪（存储、嵌入/重排后端等依赖连通性正常）；未就绪返回 `503`，供 K8s `readinessProbe` 使用，实现流量摘除与自动重启。

### 4.2 知识库管理接口

- **创建知识库**：`POST /v1/knowledge-bases`（入参：名称、描述、嵌入模型标识）
- **列出知识库**：`GET /v1/knowledge-bases`
- **删除知识库**：`DELETE /v1/knowledge-bases/{kb_id}`（级联删除文档、分块、向量、集合）

### 4.3 MCP 工具接口

- **工具名**：`search_knowledge`（详见 §3.7）
- 传输：stdio / SSE
- 语义：无状态检索，输入查询 → 输出结构化 Chunk 列表。
- 入参扩展：`timeout_ms`（可选，显式超时控制，默认 5000）、超时/重试约定见 §3.7。
- 鉴权：SSE 模式复用 REST API Key 鉴权（握手/调用时校验 `Authorization`）；stdio 模式进程内嵌默认信任。

> **已删除**：v1.0 的 `POST /v1/chat`（问答）、`GET/DELETE /v1/sessions/{session_id}`（会话管理）。

### 4.4 通用约定

- 统一错误响应结构：错误码、错误信息、请求追踪 ID。
- REST 统一鉴权（初期 API Key，预留用户体系扩展）。
- API 版本化前缀 `/v1/`。

## 五、技术选型

### 5.1 整体技术栈

| 层次 | 选型方向 | 说明 |
| --- | --- | --- |
| 开发语言 | Python | AI/检索生态成熟 |
| Web 框架 | FastAPI | 异步 + 自动 OpenAPI 文档（文档管理 REST） |
| MCP 框架 | `mcp`（官方 SDK） | 暴露 search_knowledge，支持 stdio/SSE |
| Markdown 解析 | `markdown-it-py` | AST 解析，提取标题层级与代码块 |
| JSON 解析 | `orjson` | 高性能解析/序列化，用于扁平化压平 |
| 嵌入模型 | 可插拔（OpenAI 兼容 / 本地） | 通过 .env 配置，删除 openai 硬依赖 |
| 向量+全文库 | **LanceDB** | 原生向量 + FTS 混合检索，避免双索引一致性问题 |
| 重排序 | Cross-encoder（本地/云端 API） | 可插拔，仅用于排序 |
| 元数据存储 | SQLite | 文档/分块元信息，轻量零运维 |
| 任务队列（预留） | RQ（Redis Queue） | 仅预留接口位；首版走进程内后台任务实现异步入库（默认不依赖 RQ） |
| 配置管理 | pydantic-settings / dotenv | .env 统一加载，零硬编码 |
| 容器化 | Docker + docker-compose | 一键部署 |
| 测试 | pytest | 单元 + 集成 |

### 5.2 依赖变更清单

**删除的依赖**
- `openai`（LLM 生成，改为可选嵌入适配器）
- `pypdf` / PDF 解析库（不再支持 PDF）
- `paddleocr` / OCR 库（不再支持扫描件）
- 查询改写、问答生成、会话管理相关模块

**新增的依赖**
- `mcp`（MCP Server SDK）
- `markdown-it-py`（Markdown AST 解析）
- `orjson`（高性能 JSON 处理）
- `lancedb`（向量 + 全文混合检索）

### 5.3 选型考量

- **为何用 LanceDB**：原生同库支持向量与全文（BM25）检索，写入/删除时两类索引同步更新，从根本上避免"向量库 + 独立 BM25"分离方案的**索引一致性**难题；列式存储 + 本地持久化，契合个人级零外部依赖要求。
- **为何 markdown-it-py**：CommonMark 兼容、AST 完整、可精确拿到标题层级与代码块（含语言），是递归标题继承法的可靠基础。
- **为何 orjson**：解析/序列化极快，扁平化处理大 JSON 时性能优势明显。
- **为何删除 PDF/OCR**：PDF 版式复杂、OCR 存在识别误差，是解析成功率的最大不确定源；仅保留结构化文本格式可将解析成功率提升至 ≥ 99.9%。
- **为何异步入库（202 + 轮询）**：个人级、单机、结构化格式解析快，但大文件嵌入仍可能耗时；改为上传即返回 `202`、后台异步处理并轮询，既保持"上传即提交"的简洁体验，又消除同步阻塞与 504 超时；底层进程内后台任务零外部依赖，后续可切 RQ 支撑批量。

## 六、数据模型设计

### 6.1 核心实体

**知识库（KnowledgeBase）**
- 字段：知识库 ID、名称、描述、嵌入模型标识、向量维度、活跃集合名（active_collection）、创建时间。

**文档（Document）**
- 字段：文档 ID、所属知识库 ID、原始文件名、文件类型（md/json/jsonl）、文件大小、内容哈希、上传时间、解析状态（取值 `processing` / `ready` / `failed`）、**`started_at`（解析开始时间）、`attempt_count`（入库尝试次数，用于崩溃恢复识别）**、分块数量、错误信息（若有）、**`format_version`（新增，解析器格式版本）**。

**分块（Chunk）**
- 字段：分块 ID、所属文档 ID、所属知识库 ID、块序号、文本内容、字符偏移。
- **新增字段**：
  - `content_type`：`md` / `json`
  - `section_path`：MD 块的标题路径（如 `"第三章 > 注意力机制"`），JSON 块为空
  - `json_parent_id`：JSON 块的父路径 ID（数组项/降级子块共享），MD 块为空
  - `code_language`：代码块语言标识（如 `python`），非代码块为空
- **删除字段**：`page_number`（页码，PDF 概念，已无意义）。

> 会话（Session）与对话消息（Message）实体**整体删除**（无生成、无会话）。

### 6.2 关系

- 知识库 1 — N 文档 1 — N 分块。
- JSON 数组项/降级子块通过 `json_parent_id` 形成同父归属。
- 删除知识库 → 级联删除文档、分块、向量集合；删除文档 → 级联删除其分块与向量。

## 七、非功能性设计

### 7.1 性能设计

| 指标 | 目标 | 保障措施 |
| --- | --- | --- |
| 检索响应（召回+融合+重排） | **< 500ms** | LanceDB 混合检索 + 可选重排序 + 无生成开销 |
| 10MB 文件入库可检索（processing→ready） | **< 3 秒** | 异步解析（结构化格式快，后台处理）+ 批量嵌入 |
| 解析成功率 | **≥ 99.9%** | 仅结构化文本格式，AST/orjson 稳健解析 |
| 并发检索 | ≥ 10 并发无显著退化 | 异步框架 + 检索并发 + 限流 |
| 高频相同 query 检索 | 命中缓存后延迟接近本地内存读取 | 热点查询 LRU 缓存（§3.6），跳过重排序计算 |

> **已删除指标**：流式首字延迟（无生成、无流式输出）。
>
> 引入热点查询 LRU 缓存（§3.6）后，高频相同 query 可命中缓存跳过重排序，端到端延迟进一步下降；异步入库（§3.1）使上传接口不再受 3s 入库耗时约束，上传响应即时返回 `202`。

### 7.2 可靠性与异常处理

- 解析失败：标记错误状态，记录原因，不阻塞其他文档，支持**手动重试**（幂等）。
- 嵌入调用失败：指数退避重试，超限返回明确错误。
- 索引写入失败：事务回滚，保证元数据与向量一致性（LanceDB 单库更利于一致性）。
- **进程重启后的入库恢复**：启动时扫描超时（> `INGEST_TIMEOUT_SEC * 2`）仍为 `processing` 的文档，自动置为 `failed`（`reason: "process_restarted"`），避免永久卡死，引导用户手动重试（进程内后台任务易失性的兜底）。
- 全链路结构化日志：关键节点耗时、错误堆栈、请求追踪 ID。

### 7.3 安全设计

- 密钥统一由 .env 注入，禁止硬编码与明文落盘；提供 `.env.example`，实际 `.env` 不入库。
- 文档原文与向量默认本地存储；嵌入调用仅传输必要文本片段。
- REST 鉴权（初期 API Key，预留用户体系）；**MCP SSE 模式复用同一 API Key 鉴权**（握手/调用校验 `Authorization`），stdio 模式进程内嵌默认信任。
- 上传文件类型（仅 md/json/jsonl）与大小白名单校验。
- **存储加密基础支持**：`ENABLE_STORAGE_ENCRYPTION` 开关（默认关闭）预留透明加密能力，密钥由 .env 注入，避免明文落盘（详见 §3.5）。
- **MCP 工具描述显式声明幻觉责任转移**，避免下游误用中间件结果。

### 7.4 可观测性

- 结构化日志（JSON），含请求 ID、模块、耗时、状态。
- 关键指标埋点：解析成功率、各路召回数量、融合/重排耗时、检索总耗时、按 content_type 的命中分布。
- 预留 Prometheus 指标导出接口。

## 八、部署与运维

### 8.1 本地开发

- `.env` 配置嵌入模型、重排序模型、LanceDB 路径、权重系数、并发/限流阈值等。
- 一条命令启动文档管理 REST 服务（uvicorn 热重载）。
- 一条命令启动 MCP Server（stdio 或 SSE 模式），供本地 Agent 连接。

### 8.2 Docker 一键部署

- 多阶段构建 Dockerfile，镜像精简。
- docker-compose 编排应用服务与卷映射（LanceDB 数据、SQLite、上传原文）。
- 环境变量通过 compose environment 或 .env 注入。
- 可分别暴露 REST 端口与 MCP SSE 端口。
- 容器编排支持：Dockerfile / compose 配置 `HEALTHCHECK` 调用 `/healthz`；提供 K8s 部署清单示例的 `livenessProbe`（`/healthz`）+ `readinessProbe`（`/readyz`），实现异常自动重启与未就绪时流量摘除。

### 8.3 模型升级与零停机迁移（双集合写入）

嵌入模型（Embedding）或重排序模型（Rerank）升级切换时，采用 **"双集合写入"** 策略实现零停机：

1. **新建集合**：以新模型创建新集合（collection_v2），维度按新模型确定。
2. **写新读旧**：迁移期间新数据写入新集合；查询仍走旧集合（collection_v1，**只读**），线上无感。
3. **后台重嵌**：后台批量将旧数据用新模型重新嵌入写入新集合。
4. **验证切换**：以评测集验证新集合检索质量达标后，将知识库 `active_collection` 指向新集合。
5. **异步清理**：确认稳定后，**异步清理**旧集合，释放空间。

> 全程旧集合可读、服务不中断，切换为原子操作，出问题可即时回滚指向旧集合。

### 8.4 数据备份与迁移

- LanceDB 数据目录、SQLite 文件、上传原文目录为核心数据，纳入备份。
- format_version 升级：解析器格式变更时，可按旧 format_version 批量重解析重建分块。

## 九、测试策略

### 9.1 测试分层

- **单元测试**：MD AST 解析与递归标题继承分块、JSON 扁平化与数组拆分、超长 Value 降级、动态加权融合、重排序、MCP 工具序列化。目标覆盖率 ≥ 80%。
- **集成测试**：端到端链路——上传（md/json）→ 异步入库（202 + 状态轮询）→ REST 列表/删除 → MCP `search_knowledge` 检索 → 结构化返回校验。
- **评测集构建（阶段二）**：构建查询-相关性标注集，按 content_type **分层抽样**——MD 含 3 种子类型（纯文本 / 代码块 / 嵌套列表）、JSON 含 3 种子类型（键值对 / 数组对象 / 混合），覆盖含数组、超长值的多样样本；以人工/半自动标注的 `query → 相关 Chunk` 对作为 Recall@k / MRR 量化基线与回归依据，并预留 20% 盲测集防过拟合。
- **自动调参工具**：以标注集为优化目标，对 `MD_VECTOR_WEIGHT` / `JSON_BM25_WEIGHT`、召回 top-N、重排序 top-K 等超参做网格/贝叶斯自动搜索，输出更优参数组合建议，并纳入 CI 防止权重劣化。
- **回归测试**：基于上述评测集的回归样例，防止分块/融合/权重调整引入检索质量回退。

### 9.2 质量验收项

- 解析成功率 ≥ 99.9%（md/json 标准测试集）。
- MD 分块 `section_path` 正确、代码块完整无截断、嵌套继承 ≤ 3 级（自动校验）。
- JSON 扁平化键路径正确、数组项共享 `json_parent_id`、超长 Value 正确降级。
- 检索相关性：Top-k 命中率达标（构建检索评测集量化，如 Recall@k / MRR）。
- MCP 返回结构合法：字段完整、含原文与来源元数据、不含生成内容。
- 性能指标达标（见 §7.1）。

## 十、扩展性设计

### 10.1 检索策略可插拔
召回器、融合器、重排器以接口暴露，新增算法（学习排序、多路召回）只需实现接口并注册。

### 10.2 模型可切换 + 零停机迁移
嵌入/重排序模型通过 .env 切换，配合双集合写入实现零停机升级（§8.3）。

### 10.3 格式解析器可扩展
MD/JSON 解析器以接口形式存在，未来如需扩展（如 YAML、CSV）可新增解析器实现，`format_version` 管理升级；但需坚守"结构化文本、解析可靠"原则。

### 10.4 多知识库与权限隔离
数据模型已保留知识库 ID 维度，后续可扩展用户体系与知识库级权限。

### 10.5 异步入库演进
首版即采用**异步入库（202 Accepted + 状态轮询）**，底层以 `EnqueueBackend` 抽象实现，默认进程内后台任务（零外部依赖）；大规模批量场景可平滑切换为 RQ（Redis Queue），无需改动上传/轮询接口契约。

## 十二、风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| MD 嵌套列表过深导致分块碎片化 | 检索命中质量下降 | 继承深度 ≤ 3 级，超出压平承载 |
| JSON 超长 Value 撑爆单块 | 嵌入/检索异常 | > 2000 tokens 降级为 MD 语义切分 |
| 动态权重设置不当 | MD/JSON 检索失衡 | 权重 .env 可调 + 评测集回归校准 |
| 模型升级维度不一致 | 检索错乱 | 双集合写入 + 维度一致性校验 |
| 入库大文件阻塞 | 上传接口响应慢 | 异步入库（202 + 轮询），后台任务总超时（`INGEST_TIMEOUT_SEC`，默认 30s）标记 failed；进程内任务零依赖，可平滑切 RQ；支持手动重试 |
| 热点 query 缓存失效粒度粗 | 单文档更新清空全库缓存，命中率骤降 | 缓存键引入 `doc_versions_hash` 分区驱逐；实现复杂则退化为 TTL 为主（300s）+ 主动失效为辅 |
| 存储加密开关误开 | 落盘/读取异常 | 默认关闭（`ENABLE_STORAGE_ENCRYPTION=false`），仅预留接口位，首版不影响原生读写 |
| MCP 调用超时/抖动 | Agent 检索失败 | 工具描述推荐超时（默认 5s）+ 指数退避重试（≤3 次）；Agent 可传 `timeout_ms` 显式控制 |
| 进程内后台任务易失性 | 进程崩溃/重启致文档永久卡 processing | SQLite 记录 `started_at`/`attempt_count`；启动扫描超时（>`INGEST_TIMEOUT_SEC*2`）processing 文档置 failed（reason: process_restarted）；切 RQ 后自然消失 |
| 并发上传相同文件竞态 | 重复解析/写入、资源浪费 | `(knowledge_base_id, content_hash)` UNIQUE 约束，冲突返回 409 + 已存在文档 ID，原子去重 |
| 自动调参过拟合 | 权重仅适配评测集，线上泛化差 | 评测集按 content_type 分层抽样（MD/JSON 各 3 子类型）；K-Fold 交叉验证 + 20% 盲测集 hold-out |
| MCP SSE 未授权调用 | 公网暴露被未授权检索 | SSE 握手/调用复用 REST API Key 鉴权（`Authorization`），stdio 模式默认信任 |
| 下游 Agent 误将片段当结论 | 幻觉归因不清 | MCP 工具描述显式声明责任转移 |
| LanceDB 生态较新 | 迁移/排障成本 | 存储接口抽象，可切换 Milvus/Qdrant |

## 十三、验收标准汇总

**功能正确性**
- 解析成功率 ≥ 99.9%（md/json 标准测试集）。

## 附录 B：配置项清单（环境变量）

- 嵌入相关：Endpoint、API Key、模型名、维度、批量大小。
- 重排序相关：模型/Endpoint、是否启用、候选数量上限。
- 存储相关：LanceDB 存储路径、距离度量、FTS 参数。
- 检索相关：向量召回 top-N、全文召回 top-N、重排序 top-K。
- **动态加权**：`MD_VECTOR_WEIGHT`（默认 1.1）、`JSON_BM25_WEIGHT`（默认 1.2）。
- 分块相关（MD/JSON 独立）：`MD_CHUNK_SIZE`（默认 800，字符）、`MD_CHUNK_OVERLAP`（默认 100，字符）、`JSON_CHUNK_SIZE`（默认 600，字符）、`JSON_CHUNK_OVERLAP`（默认 60，字符）、MD 嵌套继承深度上限（默认 3）、JSON 超长 Value 阈值（默认 2000 tokens）。
- 异步入库相关：`INGEST_TIMEOUT_SEC`（后台任务总超时，默认 30s）。
- 缓存相关：`QUERY_CACHE_MAXSIZE`（LRU 容量，默认 512）、`QUERY_CACHE_TTL_SEC`（缓存过期，默认 300）。
- 存储加密相关：`ENABLE_STORAGE_ENCRYPTION`（默认 false）、`STORAGE_ENCRYPTION_KEY`（密钥注入）。
- MCP 相关：`MCP_SEARCH_TIMEOUT_MS`（默认 5000，推荐超时）、重试策略（建议指数退避 ≤3 次，初始间隔 200ms）。
- 迁移相关：活跃集合名、双集合迁移开关。
- 服务相关：REST 端口、MCP 传输模式（stdio/sse）与端口、日志级别、并发限流阈值、上传大小上限、format_version、健康探针 `/healthz` 与 `/readyz`。

> 所有配置项均通过 .env 注入，提供 `.env.example`，禁止硬编码。
